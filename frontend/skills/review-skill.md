# review-skill — 双盲评审

## 概述
使用两个独立视角（用户视角 + 工程视角）并行审查交付物，通过共识表裁决。

## 评审流程

### Step 1: 提交评审
提供需要审查的内容（代码/文档/配置等）。

### Step 2: 双视角审查

**视角 A — 用户视角（挑剔用户）**
- 功能是否完整？
- 错误信息是否友好？
- 使用流程是否顺畅？
- 边界情况是否处理？

**视角 B — 工程视角（严格工程师）**
- 代码质量如何？
- 是否有安全风险？
- 性能是否可接受？
- 是否易于维护？

### Step 3: 共识表裁决

| 裁决 | 条件 |
|------|------|
| **FIX** | 两个视角都发现问题，或任何 CRITICAL 级别 |
| **TRIAGE** | 仅一个视角发现问题（非严重） |
| **DISMISS** | 被另一个视角反驳，或仅 SUGGESTION 级别 |

### Step 4: 修复验证（如需）
FIX 项修复后重新评审（最多 2 轮）。

## 问题严重级别

| 级别 | 说明 |
|------|------|
| CRITICAL | 功能不可用 / 数据丢失 / 安全漏洞 |
| MAJOR | 功能异常 / 体验差 |
| MINOR | 小问题 / 不影响主流程 |
| SUGGESTION | 优化建议 / 锦上添花 |

## 使用方式
在需要评审时，按以下格式描述评审对象：
1. 评审对象（文件/功能/文档）
2. 评审标准（可选，默认全面评审）
3. 重点关注（可选，如安全性、性能）
